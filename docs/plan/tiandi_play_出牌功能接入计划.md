### 3.2 本轮已确认的定稿点

- 跟牌时允许：
  - 同牌型同结构压制
  - 或任意炸弹 / 王炸压制
- 四带两单与三带二（即三带一对）等异型允许比较。
  - 比较时只看主干：
    - 四带类看四张主牌点数
    - 三带类看三张主牌点数
  - 尾巴不参与大小比较
  - 三带类与四带类直接按主牌点数互比，不因主干张数不同增加额外权重
- 飞机带牌时，附带牌允许包含赖子原牌。
  - 该用法虽然通常不优，但规则层允许。
- 赖子多解冲突时：
  - 总是返回全部可能
  - 若多个候选中存在炸弹解释，炸弹优先级最高
- 单出赖子时，不需要单独区分“解释牌”展示。
  - 前端直接展示该赖子牌本体即可
  - 如后端有解析信息，也只作为附加说明，不替换主展示
- “不出”仅在桌面已有主牌时可用。
  - 不允许首出空动作
- 一轮所有其他玩家都 `pass` 后：
  - 下一手必须由上一手赢家重新首出任意牌

- 一份正式规则确认稿。
- 一份“实现映射表”，把自然语言映射为后端字段与前端展示字段。
- 一份“歧义决议表”，列明每条争议规则的最终选择。
- 一份“典型牌例表”，作为测试用例与 fixture 来源。

### 3.4 建议的规则落地原则

- 后端是唯一判型和比较事实源。
- 前端不做牌型判断，只做：
  - 选牌
  - 提交
  - 显示后端解析结果
- 赖子多解不由前端猜测，必须由后端统一返回候选解列表与标准选择结果。
- 当候选解中存在炸弹解释时，后端标准选择必须优先落到炸弹解释。

### 3.5 已确认规则决议

- 跟牌时，除同型压制外，允许任意炸弹或王炸压制。
- 四带两单与三带二（即三带一对）等异型允许比较，但只比较主干，不比较尾牌。
- 三带类与四带类直接按主牌点数互比，不因主干张数不同增加额外权重。
- 飞机带牌允许携带赖子原牌。
- 赖子多解必须全部返回，且炸弹解释优先级最高。
- 单出赖子前端直接展示原牌本体，不额外切换成“解释牌”。
- `pass` 仅在桌面已有主牌时合法。
- 两家连续 `pass` 后，上一手出牌成功者重新获得首出权。

### 3.6 T1 规则确认结果

#### A. 正式规则清单

- 首出：
  - 当前持有出牌权的玩家可出任意合法牌型。
- 跟牌：
  - 默认可用同型同结构压制。
  - 也允许任意炸弹压制普通牌。
  - 王炸压制全部其他牌型。
- 带牌类比较：
  - 只比较主干牌点。
  - 尾牌只用于结构合法性，不参与大小。
  - 已明确覆盖：
    - 四带两单
    - 四带两对
    - 三带一
    - 三带二
- 连续类比较：
  - 顺子、连对、飞机先比较长度/组数。
  - 长度相同再比较主干最高牌点。
- 赖子规则：
  - 赖子不能替代大小王。
  - 单独打出赖子时，只按赖子本体处理。
  - 与其他牌组合时可替代非王点数。
  - 飞机带牌允许带赖子原牌。
- 多解规则：
  - 后端必须返回全部可行解析。
  - 若存在炸弹候选，标准优先结果落到炸弹候选。
- `pass` 规则：
  - 只有桌面已有当前主牌时才能 `pass`。
  - 首出方不能 `pass`。
- 清轮规则：
  - 一家成功出牌后，若另外两家连续 `pass`，则当前轮结束。
  - 上一手成功出牌者重新获得首出权。

#### B. 歧义决议表

| 主题 | 最终决议 | 备注 |
|---|---|---|
| 跟牌压制 | 同型同结构压制，或任意炸弹 / 王炸跨型压制 | 后端统一判断 |
| 带牌类比较 | 只比主干，不看尾巴 | 尾牌只校验结构 |
| 飞机附带牌 | 允许包含赖子原牌 | 不做额外限制 |
| 赖子多解 | 全部返回，炸弹优先 | 既保留候选，也给标准结果 |
| 单出赖子展示 | 前端展示原牌本体 | 解析信息仅作附加展示 |
| `pass` 可用条件 | 桌面已有主牌时才可用 | 不允许空动作 |
| 清轮后首出权 | 上一手赢家重新首出 | 不沿用顺时针强制跟进 |

#### C. 自然语言规则到后端字段的映射建议

- `牌型` -> `kind`
- `主干比较点` -> `coreRank`
- `主干规模` -> `coreSize`
- `连续长度/组三长度` -> `sequenceLength`
- `带牌结构` -> `attachmentShape`
- `是否炸弹/火箭` -> `isBomb` / `isRocket`
- `是否使用赖子` -> `usesLaizi`
- `赖子解析明细` -> `resolution`
- `全部候选解` -> `resolutionCandidates`
- `标准优先解` -> `preferredResolution`
- `当前桌面主牌` -> `currentTrick`
- `上一手成功出牌者` -> `lastPlaySeat`
- `当前是否可 pass` -> `availableActions` 是否包含 `pass`

#### D. 典型牌例列表

- `333` 对 `444`：
  - 同型比较，`444` 胜
- `333 + 44` 对 `444 + 55`：
  - 三带二比较，只看三张，`444` 胜
- `4444 + 5 + 6` 对 `333 + 77`：
  - 带牌类跨型比较，只看主干，`4444` 主干大于 `333`
- `34567` 对 `45678`：
  - 顺子同长度比较，`45678` 胜
- `333444 + 5 + 6` 对 `444555 + 7 + 8`：
  - 飞机带单，同组数比较最高三张，后者胜
- `7777` 对 `333444`：
  - 炸弹跨型压制普通牌
- `BlackJoker + RedJoker` 对 `AAAAA`：
  - 王炸压制 5+ 炸
- `天赖 + 7 + 7`：
  - 可解析为 `777`
- `天赖 + 地赖 + 7 + 7`：
  - 需要返回全部可行解析；若可成炸弹，则炸弹为标准优先解
- 单出 `heart-A`，且 `A` 为赖子：
  - 前端展示 `heart-A`
- 桌面无主牌时尝试 `pass`：
  - 非法
- 玩家 A 出牌后，B/C 连续 `pass`：
  - A 重新首出

## 4. 后端与前端插入点设计

### 4.1 后端插入点

- 领域层新增建议
  - `backend/internal/tiandi/play/`
  - 目标：承载出牌阶段核心领域逻辑

  - `analyze.go`

  - `compare.go`
    - 同型比较、炸弹压制
  - `laizi_resolve.go`
    - 赖子替代求解、标准解选择
  - `turn.go`
    - 出牌、跟牌、过牌、轮次结束
  - `result.go`
    - 胜负判定、结束态

  - `backend/internal/tiandi/fsm/machine.go`

  - `backend/cmd/tiandi-server/main.go`

## 8. 风险与控制

- 风险 1：规则未定稿就开始写后端，导致返工。
  - 控制：先完成 `S2`，再进入实现。
- 风险 2：赖子多解策略不清晰，导致前后端协议反复变化。
  - 控制：先定标准解策略，再定 DTO。
- 风险 3：前端自己做判型，和后端产生分叉。
  - 控制：前端只做选牌与显示，不做规则判断。
- 风险 4：测试数据不可复用，后续维护成本高。
  - 控制：夹具统一来源于“典型牌例表”。
